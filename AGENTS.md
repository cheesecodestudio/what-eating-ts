# What Eating TS — Agent Instructions

Meal-planning SPA for home cooks. Backend computes ingredient availability; frontend lets users manage inventory and dishes.

## Project Documentation (read before making changes)

The `docs/` folder is the source of truth for product decisions. **Read relevant docs before starting any non-trivial task.**

| File | When to read |
|---|---|
| [docs/PRD.md](docs/PRD.md) | Before adding features, routes, modules, or changing data models. Contains requirements, roadmap, open questions, and known gaps. |

### Keeping docs current
After completing work that affects the product scope, **update `docs/PRD.md`** as part of the same task:
- New module implemented → move it from §4.2 (out of scope) to §4.1 (implemented) and update §6 (system state).
- New table or endpoint → update §6.2 / §6.3.
- Open question resolved → remove or answer it in §13.
- New risk identified → add to §10.
- Do **not** update the PRD for internal refactors, style fixes, or bug fixes that don't change product behavior.

## Build & Run

```bash
# Full-stack dev (runs both concurrently)
npm run dev

# Frontend only (port 5173)
npm run dev:client

# Backend only (port 3001)
npm run dev:server

# Production build (frontend)
npm run build
```

Backend requires `server/.env`:
```
SUPABASE_URL=...
SUPABASE_SERVICE_ROLE_KEY=...
```

Vite proxies `/api/*` → `http://localhost:3001`.

## Architecture

- **Frontend**: React 18 + TypeScript + Vite + Tailwind + React Router v7
- **Backend**: Express 5 + TypeScript + Supabase JS (`@supabase/supabase-js`)
- **Database**: Supabase PostgreSQL — no RLS (service role key used by server)
- **Schema**: [scripts/supabase-migration.sql](scripts/supabase-migration.sql)

Active DB tables: `ingredients`, `dishes`, `dish_ingredients`.  
`plates` and `plate_ingredients` are defined in `server/src/routes/plates.ts` but **not mounted** in `server/src/index.ts`.

## Key Conventions

### Naming
- DB columns: `snake_case` (`in_stock`, `food_group`, `create_date`)
- API/frontend types: `PascalCase` (`InStock`, `FoodGroup`, `CreateDate`)
- IDs: prefixed strings — `I_<timestamp>_<hex>` (ingredients), `D_...` (dishes), `DI_...` (dish-ingredients), generated by `server/src/utils.ts#generateId`

### Frontend Types
Defined in `src/types/index.ts` and enums in `src/enums/Enums.ts`.  
- `FoodGroup`: `Dairy | Grain | Protein | Vegetable | Fruit | Fat | Sugar`
- `DishType`: `Every | Breakfast | Meat | No Meat | Fast Food`
- `UnitOfMeasure`: `Grams | Ml | Unit | Cup`
- Ingredients support dual food groups (`FoodGroup` + `FoodGroups: FoodGroup[]`); backend normalizes both in `resolveFoodGroupPayload`.

### Backend Routes
Each route file in `server/src/routes/` exports an Express router with CRUD operations.  
Mount pattern in `server/src/index.ts`:
```ts
app.use('/api/ingredients', ingredientsRouter)
app.use('/api/dishes', dishesRouter)
```
Always add `mapRow` / `mapDish` functions to convert DB rows → API shape before responding.

### CanMake Logic
Computed by `server/src/routes/dishes.ts#mapDish`: true only if every `dish_ingredient` has its linked ingredient `in_stock = true`. A dish with no ingredients is always `false`.

### Frontend Data Fetching
Use `useApi<T>(url)` for reads (returns `{ data, loading, error, refetch }`).  
Use `apiPost`, `apiPut`, `apiDelete` from `src/hooks/useApi.ts` for mutations; always call `refetch()` afterwards.

## Common Pitfalls

- `server/.env` is not committed — agents cannot run the server without it.
- `plates.ts` route exists but is intentionally unmounted; do not assume it's active.
- `food_groups TEXT[]` was added as a migration column; `food_group TEXT` (singular) also exists for backward compatibility — always write both via `resolveFoodGroupPayload`.
- No authentication layer exists; all API endpoints are open.
- No automated tests currently exist in the project.
